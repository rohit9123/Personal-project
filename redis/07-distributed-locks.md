# 07 — Redis Distributed Locks: SETNX, Redlock & Fencing

> **Why this is Topic 7:** In a distributed backend system, standard Java lock primitives (like `synchronized` or `ReentrantLock`) only work within a single JVM process. If you have 5 instances of an order microservice running behind a load balancer, they cannot coordinate using local locks. To prevent race conditions—like a user submitting two identical buy orders at the same millisecond—you need a **Distributed Lock** that spans the network. SDE2 interviewers will test your ability to implement a secure single-instance lock, explain the Redlock algorithm, articulate Martin Kleppmann's famous distributed systems critique of Redlock, and explain the architectural fix: fencing tokens.

---

## 1. WHAT

A **Distributed Lock** is a mechanism to ensure mutual exclusion across multiple servers accessing a shared resource. 

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  Instance A  │     │  Instance B  │     │  Instance C  │
└──────┬───────┘     └──────┬───────┘     └──────┬───────┘
       │                    │                    │
       │ 1. SET NX PX (Win) │ 2. SET NX PX (Fail)│ 3. SET NX PX (Fail)
       └──────────────┐     │                    │
                      ▼     ▼                    ▼
               ┌──────────────────────────────────┐
               │         Redis Lock Key           │
               │  "lock:user:100" -> "unique-id"  │
               └──────────────────────────────────┘
```

Redis supports two primary patterns:
1. **Single-Instance Lock:** Utilizing the atomic `SET key value NX PX` command on a single master node, released via a Lua script.
2. **Multi-Instance Lock (Redlock):** Acquiring locks across multiple independent Redis nodes to tolerate single-node hardware failures.

---

## 2. WHY (the trade-offs & critiques)

### 2.1 The Danger of Naive Locks
A naive implementation like `SETNX lock:key 1` (without an expiration TTL) is highly vulnerable. If the application instance crashes or loses network connectivity immediately after acquiring the lock, the lock key will remain in Redis forever, creating a permanent **deadlock**. 

To solve this, we must enforce a TTL (`PX`). However, introducing a TTL introduces the risk of the lock expiring *while the thread is still executing*, leading to race conditions.

---

### 2.2 Martin Kleppmann’s Critique of Redlock
In 2016, distributed systems researcher Martin Kleppmann published a critique of Redis's multi-instance **Redlock** algorithm, arguing that it is unsafe for systems where correctness is non-negotiable.

#### 1. Clock Drift / NTP Jumps
Redlock calculates lock acquisition time using local system clocks. If Node 3's clock jumps forward due to an NTP sync while Instance A is acquiring a lock:
1. Instance A acquires the lock on Nodes 1, 2, and 3.
2. Node 3's clock jumps forward, causing Node 3 to immediately expire the lock key.
3. Instance B now successfully acquires the lock on Nodes 3, 4, and 5 (a majority).
4. **Failure:** Both Instance A and Instance B believe they hold the lock simultaneously, violating mutual exclusion.

#### 2. Process Pauses (GC/VM Pauses)
If Instance A acquires the lock, and then suffers a Stop-the-World garbage collection (GC) pause or a VM freeze:

```
Instance A ──► [Acquires Lock] ──► [GC Pause Starts (15 seconds)]
                                       │ (Lock expires in Redis after 10s)
Instance B ──► [Acquires Lock] ◄───────┘
Instance B ──► [Writes to Database]
Instance A ──► [GC Pause Ends] ──► [Writes to Database (Corrupts Instance B's write!)]
```

#### The Architectural Solution: Fencing Tokens
To guarantee safety under process pauses, the database must cooperate using **Fencing Tokens** (monotonically increasing numbers).

```
Client A ──► [Gets Lock] ──► [Token = 83] ──► [GC Pause Starts]
Client B ──► [Gets Lock] ──► [Token = 84] ──► [Writes to DB (DB records MaxToken = 84)]
Client A ──► [GC Ends]   ──► [Tries write with Token = 83] ──► DB Rejects (83 < 84)
```

## 3. HOW (the internals)

### 3.1 Single-Instance Lock Internals

#### 1. Safe Acquisition
To acquire a lock, you must use the atomic `SET` command with parameters:
```redis
> SET lock:user:100 "unique_request_uuid" NX PX 30000
```
* **`NX`:** Only set the key if it does not already exist. This ensures only one client wins.
* **`PX 30000`:** Sets a millisecond TTL (30 seconds) to prevent permanent deadlocks.
* **`unique_request_uuid`:** A unique identifier generated by the client. This is crucial for **safe release**.

#### 2. Safe Release
If Client A acquires the lock, takes 35 seconds to finish (longer than the 30-second TTL), the lock expires. Client B then acquires the lock. 
If Client A finishes at second 35 and calls a simple `DEL lock:user:100`, **it will delete Client B's lock!**

To prevent this, the release must be executed via an **atomic Lua script** that only deletes the key if the value matches the client's unique UUID:

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

---

### 3.2 Redlock Algorithm Details
To tolerate master node failures, Redlock uses $N$ independent Redis masters (usually 5):
1. **Timestamp $T1$:** The client gets the current time.
2. **Acquisition Attempt:** The client attempts to set the lock key (using `NX PX`) on all 5 nodes sequentially, using a tiny socket timeout (e.g. 5–50ms) to avoid getting blocked by a crashed node.
3. **Majority Check:** The client checks if it acquired the lock on a **majority** ($\ge 3$) of nodes, and that the total time taken ($T2 - T1$) is less than the lock TTL.
4. **Lock Granted:** If both conditions are met, the lock is acquired. The actual time remaining on the lock is:
   $$\text{Validity Time} = \text{Initial TTL} - \text{Acquisition Duration} - \text{Clock Drift Allowance}$$
5. **Rollback:** If it fails to acquire the majority, it immediately sends a release Lua script to all 5 nodes to clean up any partial locks.

---

### 3.3 Redisson's Watchdog Mechanism (Java Production Standard)
In Java, manually managing lock renewals is complex. The library **Redisson** handles this automatically using a **Watchdog timer thread**:

```
Client ──► [Acquires Lock (Default TTL 30s)]
                 │
                 ├─► [Spawns Watchdog Thread]
                 │         │ (Every 10s: Checks if client is still processing)
                 │         └─► [Sends command to Redis: Extend TTL back to 30s]
                 ▼
[Client Finishes / Releases Lock] ──► Watchdog stops.
```

If the client application crashes, the watchdog thread dies. The lock key naturally expires in Redis after 30 seconds, preventing deadlocks.

## 4. CODE / EXAMPLES (Exclusively Java)

### 4.1 Secure Single-Instance Lock using Jedis

```java
import redis.clients.jedis.Jedis;
import redis.clients.jedis.params.SetParams;
import java.util.Collections;
import java.util.UUID;

public class JedisDistributedLock {
    private static final String RELEASE_LUA_SCRIPT = 
        "if redis.call('get', KEYS[1]) == ARGV[1] then\n" +
        "    return redis.call('del', KEYS[1])\n" +
        "else\n" +
        "    return 0\n" +
        "end";

    public boolean executeWithLock(String lockKey, long leaseTimeMs, Runnable task) {
        String uniqueId = UUID.randomUUID().toString();
        
        try (Jedis jedis = new Jedis("localhost", 6379)) {
            // 1. Acquire Lock atomically using NX and PX
            SetParams params = SetParams.setParams().nx().px(leaseTimeMs);
            String result = jedis.set(lockKey, uniqueId, params);
            
            if ("OK".equals(result)) {
                try {
                    task.run(); // Execute task under lock protection
                    return true;
                } finally {
                    // 2. Release Lock safely using Lua script
                    jedis.eval(
                        RELEASE_LUA_SCRIPT, 
                        Collections.singletonList(lockKey), 
                        Collections.singletonList(uniqueId)
                    );
                }
            }
            return false; // Failed to acquire lock
        }
    }
}
```

### 4.2 Single-Instance Lock using Spring Data Redis

```java
import org.springframework.data.redis.core.StringRedisTemplate;
import org.springframework.data.redis.core.script.DefaultRedisScript;
import org.springframework.data.redis.core.script.RedisScript;
import java.util.Collections;
import java.util.UUID;
import java.util.concurrent.TimeUnit;

public class SpringRedisLock {
    private final StringRedisTemplate redisTemplate;
    private final RedisScript<Long> releaseScript;

    public SpringRedisLock(StringRedisTemplate redisTemplate) {
        this.redisTemplate = redisTemplate;
        
        String scriptSource = 
            "if redis.call('get', KEYS[1]) == ARGV[1] then\n" +
            "    return redis.call('del', KEYS[1])\n" +
            "else\n" +
            "    return 0\n" +
            "end";
        this.releaseScript = new DefaultRedisScript<>(scriptSource, Long.class);
    }

    public boolean executeWithLock(String lockKey, long timeoutMs, Runnable task) {
        String uniqueId = UUID.randomUUID().toString();
        
        // 1. Acquire Lock (SET NX PX equivalent in Spring)
        Boolean acquired = redisTemplate.opsForValue().setIfAbsent(
            lockKey, 
            uniqueId, 
            timeoutMs, 
            TimeUnit.MILLISECONDS
        );
        
        if (Boolean.TRUE.equals(acquired)) {
            try {
                task.run();
                return true;
            } finally {
                // 2. Safe Release via Lua Script
                redisTemplate.execute(
                    releaseScript, 
                    Collections.singletonList(lockKey), 
                    uniqueId
                );
            }
        }
        return false;
    }
}
```

### 4.3 Production Lock using Redisson (With Auto-Renewal Watchdog)

```java
import org.redisson.Redisson;
import org.redisson.api.RLock;
import org.redisson.api.RedissonClient;
import org.redisson.config.Config;
import java.util.concurrent.TimeUnit;

public class RedissonLockService {
    private final RedissonClient redissonClient;

    public RedissonLockService() {
        Config config = new Config();
        config.useSingleServer().setAddress("redis://127.0.0.1:6379");
        this.redissonClient = Redisson.create(config);
    }

    public void processCriticalTask(String lockKey) {
        RLock lock = redissonClient.getLock(lockKey);
        
        try {
            // Acquire lock. 
            // Wait up to 5 seconds to get the lock. 
            // Do NOT specify leaseTime -> triggers Redisson's Watchdog!
            boolean acquired = lock.tryLock(5, TimeUnit.SECONDS);
            
            if (acquired) {
                try {
                    // Task execution (Watchdog extends lock TTL automatically every 10s)
                    executeLongTask(); 
                } finally {
                    lock.unlock(); // Release lock
                }
            } else {
                System.out.println("Could not acquire lock, task skipped.");
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }

    private void executeLongTask() {
        try {
            Thread.sleep(15000); // 15 seconds task
        } catch (InterruptedException ignored) {}
    }
}
```

## 5. INTERVIEW ANGLES

### Q: Why must a distributed lock release command use a Lua script?
**A:** If you release a lock using a simple delete command (`DEL lock_key`), you create a race condition.
If Client A acquires a lock with a 30s TTL, but takes 35s to complete, the lock expires. Client B immediately acquires the lock. 
At second 35, Client A finishes and runs `DEL lock_key`. Without checking ownership, Client A will delete Client B's lock while Client B is still executing its critical section.
**The Lua script** guarantees atomicity: it reads the lock key, compares the stored value against Client A's unique token (UUID), and deletes the key *only* if they match.

### Q: Explain Martin Kleppmann's critique of the Redlock algorithm.
**A:** Martin Kleppmann identified two major vulnerabilities in Redlock:
1. **Clock Drift:** Redlock assumes system clocks across nodes are synchronized. If Node 3 suffers an NTP clock jump forward, it will expire its lock key early. Client B can then acquire the lock on Nodes 3, 4, and 5 (a majority) while Client A still holds the lock on Nodes 1, 2, and 3, violating mutual exclusion.
2. **Process Pauses:** If Client A acquires the lock and then suffers a long Garbage Collection (GC) pause or VM freeze, the lock TTL will expire in Redis. Client B can then acquire the lock. When Client A wakes up from the GC pause, it has no way of knowing its lock has expired, and will proceed to write to the database, corrupting Client B's database changes.

### Q: How do Fencing Tokens resolve the GC pause problem?
**A:** Fencing tokens are monotonically increasing integers generated by the lock coordinator (or database sequence). 
1. When Client A acquires the lock, it receives `Token = 83`.
2. Client A pauses due to a GC cycle, and its lock expires.
3. Client B acquires the lock and receives `Token = 84`.
4. Client B writes to the database. The database records that the latest processed write has `Token = 84`.
5. Client A wakes up and attempts to write using `Token = 83`.
6. The database rejects Client A's write because `83 < 84`, protecting data integrity.

### Q: What is Redisson's Watchdog mechanism, and when should you avoid setting a lock lease time?
**A:** If you call Redisson's lock method **without** specifying a lease time (e.g. `lock.tryLock(wait, TimeUnit)`), Redisson initiates its **Watchdog thread**. 
The watchdog runs in the background and regularly checks if the client thread is still alive. If so, it sends an asynchronous command to Redis to extend the lock TTL back to 30 seconds every 10 seconds.
**Warning:** If you explicitly pass a lease time (e.g., `lock.tryLock(wait, 15, TimeUnit.SECONDS)`), Redisson **disables the watchdog**. The lock will expire strictly after 15 seconds, exposing you to race conditions if your task runs slow.

## 6. ONE-LINE RECALL CARDS

*   Naive `SETNX` locks without expiration TTL are vulnerable to permanent **deadlocks** if the client crashes.
*   Secure locks use **`SET key UUID NX PX time`** to set a TTL and lock ownership identifier atomically.
*   Releasing a lock requires an **atomic Lua script** that verifies UUID ownership before deleting the key.
*   **Redlock** attempts to acquire locks on $N$ independent masters sequentially, requiring a majority check ($\ge 3$ of 5).
*   Kleppmann's critique proves Redlock is unsafe under **NTP clock drift** and **long client process pauses**.
*   **Fencing tokens** (monotonically increasing IDs) prevent stale clients from overwriting database changes.
*   Redisson's **Watchdog** automatically renews lock TTLs back to 30s every 10s as long as the client thread is active.
*   Specifying a custom **lease time** in Redisson disables the watchdog, making the lock expire strictly at that time.
